开了个新坑《帧同步卡牌手游》，逐步认识到帧同步技术。说到帧同步那就绕不开——**定点数**。下文讨论下定点数的来龙去脉、几种实现方式，以及为什么推荐用位移放大理一遍。

## 为什么不能用 float

帧同步的核心思路是：客户端之间**只同步操作指令**，各自本地跑一套完全相同的逻辑，从而得到完全相同的结果。这就要求所有客户端的计算**逐位一致**——只要有一台机器的某个数值差了一点点，随着帧数推进就会被不断放大，最后两边的战斗画面完全对不上，也就是所谓的“不同步”。

而 `float` 恰恰做不到跨平台完全一致，原因主要有几个：

- **IEEE 754 的实现差异**：虽然大多数平台都遵循 IEEE 754，但不同硬件、编译器在舍入和精度处理上仍有细微差别。
- **编译器优化**：不同编译器可能重排浮点运算顺序以提升性能，而重排本身就可能引入微小误差。
- **硬件差异**：CPU、GPU 处理浮点的行为不完全一致。
- **数学库不同**：`sin`、`cos`、`sqrt` 等函数在 Windows、Linux、macOS 上可能来自不同的数学库（如 glibc），默认精度和舍入模式都可能不同。

单精度 `float` 大约只有 7 位十进制精度，这点误差在单机上无所谓，但在帧同步这种“蝴蝶效应”场景里就是致命的。

> [!NOTE] 注
> 帧同步要的不是“算得准”，而是“所有机器算得**一模一样**”。float 保证不了后者，所以要换成定点数。

## 什么是定点数

定点数是一种**小数点位置固定不变**的数值表示方法。相比浮点数，它精度固定、运算速度快，而且结果完全可复现——只要输入相同，任何平台上的输出永远相同。

实际做帧同步时，几乎没有人会真的去保留小数位来运算。常见做法是**保留 0 位小数，整个逻辑层完全用整型来算**，这样才能把误差压到最小。定点数的价值就在这里：

- 解决浮点特性导致的多平台不同步。
- 安全可靠，多平台计算/同步结果完全一致。
- 数据源相同则结果永远相同，方便做战斗逻辑的模拟和回放。
- 不只用于帧同步，状态同步等对精度敏感的场景也能用。
- 运算速度高于浮点数。

## 三种实现方式

### 1. 小数定位法

直接固定浮点数的小数位，比如只保留一两位。加减看起来没问题，但一旦涉及乘除这类复杂运算，因为底层还是浮点，就会产生更多小数，只能靠技术手段截断——牺牲了准确率。基本没人这么用。

### 2. 乘法放大入整法

确定一个放大因子（比如 `1000`），所有要转成定点数的数值都乘以这个因子再转成整数。之后的计算全是整型运算，彻底排除浮点误差；最后要渲染时再除回放大因子交给渲染层。

思路没问题，但细节容易翻车：放大 1000 倍时，如果处理不当仍会丢失精度（后面会看到具体例子）。

### 3. 位移运算入整法（推荐）

提前确定一个**放大次幂**，比如 10，那么每个数值转定点数时都左移 10 位（`<< 10`），相当于放大 `2^10 = 1024` 倍。因为位移只能用于整型，天然契合定点数；又因为它是 2 的次幂运算，效率极高。

这是我最推荐的实现方式，原因下面细说。

## 位移运算的原理

位移分为 `<<`（左移）和 `>>`（右移），可以类比乘除：左移像乘法，右移像除法。区别在于位移是 **2 的次幂**运算，而乘除是普通的相乘相除。

以 `1` 为例：

```plain
1 << 1  = 2      // 2^1
1 << 2  = 4      // 2^2
1 << 3  = 8      // 2^3
1 << 10 = 1024   // 2^10
```

所以左移 1 位相当于乘 2，左移 10 位相当于乘 1024。

## 隐式转换与显式转换

在 C# 里实现定点数结构体时，会大量用到 `implicit` 和 `explicit` 这两个和 `operator` 搭配的关键字，用来控制 `FixInt` 与其他类型之间怎么互转。

**隐式转换（implicit）**：由系统自动完成，不会改变精度、不会引发异常，通过 `=` 直接赋值即可触发。

```csharp
// 在 float 值赋给 FixInt 时触发
public static implicit operator FixInt(float v)
{
    return new FixInt(v);
}
```

```csharp
int intNum = 1;
long longNum = intNum;   // 隐式转换
FixInt fixNum = intNum;  // 隐式转换
```

**显式转换（explicit）**：也就是平时说的强制转换，必须通过 `(类型)` 显式指定目标类型，可能引发异常或精度丢失。

```csharp
// 在 FixInt 通过 (float) 转成 float 时触发
public static explicit operator float(FixInt v)
{
    return v.RawFloat;
}
```

```csharp
double dNum = 1.2321412989585;
float fNum = (float)dNum;  // 强制转换
int intNum = (int)dNum;    // 强制转换
```

区别很简单：**加了 `(类型)` 就是显式（强制）转换，直接用 `=` 赋值就是隐式转换。**

## 为什么位移比乘法更稳

位移比乘除快是毋庸置疑的：CPU 的移位指令一般单周期就能执行完，而乘除是多周期指令（移位约 2 个机器周期，乘除约 4 个），所以更省时间。

但更关键的是**在大数据量下位移还更准**。下面这段代码同时用“乘法放大 1000 倍”和“位移放大 1024 倍”来算同一个逻辑：

```csharp
void Start()
{
    int a = 1000;

    float value1 = (long)(a * 1000 * 1.0f / 3400);
    Debug.Log("乘法扩大 value1:" + value1);

    float value2 = (long)((a << 10) * 1.0f / 3400);
    Debug.Log("移位运算 value2:" + value2);

    Debug.Log("result float:" + (value1 / 100));
    Debug.Log("result float:" + (value2 / 100));
    Debug.Log("result int:" + (int)(value1 / 100));
    Debug.Log("result int:" + (int)(value2 / 100));
}
```

两种方式最终强转成 `int` 后，结果会差一个整数。问题出在 C# 的 `int` 显式转换是**向下取整**的：本该是 `3` 的 `2.94`，强转后变成了 `2`。这种偏差在战斗系统里非常致命。

有两种办法避免：

- 转 `int` 时用 `Mathf.Round()` 做四舍五入，而不是直接强转。
- 干脆用 **1024**（2 的次幂）而不是 1000 作为放大基准，从源头避免偏差。

> [!WARNING] 为什么偏偏是 1024
> 计算机里 `1KB = 1024 字节` 而不是 1000，就是因为 2 的次幂在底层运算、图形压缩格式里都有巨大优势。只要放大基数是 2 的次幂，配合位移运算，计算机就能保证结果既准又快。因为我们是要给数字做一个扩大处理，来解决浮点数问题，所以我们在运算的时候，1000和1024 是有差别的，这个时候我们可以把一个int数值理解为一个byte，而1024个byte才代表一个值，而我们把一个字节扩大1000倍很显然是不准的，这种不准在数值计算量大的时候就会显现出来，会造成数值误差。所以 1024 才是准确的一个倍数，我们使用这个值进行扩大之后，能够保证我们在计算中不会出现偏差。

## 使用时的注意事项

- **定点数转成浮点数后，这个浮点值就不能再参与帧同步的任何计算了。** 帧同步逻辑里全程用整型算，只有在把结果交给 UI / 渲染表现时，才转成浮点数。
- 负数用补码表示，右移操作对负数有误差，稳妥做法是**先转成正数移位、再转回负数**。
- 定点数做乘除时容易溢出：乘法结果要**除以**放大倍数，除法结果要**乘以**放大倍数，中间过程注意用更大的整型（如 `long`）承接，避免溢出。

## 总结

定点数的本质，就是把“靠不住”的浮点运算，换成“到处都一样”的整型运算：

```plain
转入定点数：数值 << 10（放大 1024 倍）
逻辑层：全程整型运算，结果完全可复现
转出表现层：右移或除回 1024，再交给渲染
```

只要保证放大基数是 2 的次幂、并用位移代替乘除，就能兼顾**精度一致**和**运算效率**。这也是为什么几乎所有成熟的帧同步方案，最后都会落到一套自己的定点数（`FixInt` / `FixVector`）数学库上。
